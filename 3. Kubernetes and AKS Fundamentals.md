- Extends the az aks command group with:
- New subcommands
- New flags/parameters
- Support for new AKS features before GA

you get:
- Extra commands under `az aks`, like:
- `az aks mesh`, `z aks approuting`, or new `az aks nodepool` options (depending on version).
- Extra parameters on existing commands, e.g.:
- New flags on `az aks create`, `az aks update`, `az aks nodepool add`, etc.
- Access to preview-only features that are not yet in the stable CLI.
The docs often say things like:
“This parameter is only available with the `aks-preview` extension.”

### Understand multi-container Pod design patterns (e.g. sidecar, init and others)

* A Pod can contain one or more containers that share the same network namespace and storage volumes.
* You can think of a Pod as a small VM where apps can communicate with each other and share data.

### Sidecar containers

* The design pattern of running multiple containers in a Pod, is often referred to as the "sidecar pattern".
* This is useful when you have a main application container with one or more containers that provide additional functionality such as logging, monitoring, caching, networking, etc.

**The following command to create a simple application that writes logs to a file and a sidecar container that reads the logs from the file and outputs them to stdout.**

```bash
kubectl apply -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: sidecardemo
spec:
  replicas: 1
  selector:
    matchLabels:
      app: sidecardemo
  template:
    metadata:
      labels:
        app: sidecardemo
    spec:
      containers:
        - name: mylogger
          image: alpine:latest
          command: ['sh', '-c', 'while true; do date >> /opt/logs.txt; sleep 1; done']
          volumeMounts:
            - name: data
              mountPath: /opt
      initContainers:
        - name: logshipper
          image: alpine:latest
          restartPolicy: Always
          command: ['sh', '-c', 'touch /opt/logs.txt && tail -F /opt/logs.txt']
          volumeMounts:
            - name: data
              mountPath: /opt
      volumes:
        - name: data
          emptyDir: {}
EOF
```

* Typically, we use YAML manifest files with `kubectl apply -f` command.
* Here we are using the `kubectl apply -f -` command to create the resources directly.

**This will create a Pod with two containers: `mylogger` and `logshipper`.**

* **mylogger** writes the current date to a file every second
* **logshipper** reads the file and outputs the contents to stdout

Run the following command to see the logs of the logshipper container.

```bash
# get the name of the sidecardemo Pod
POD_NAME=$(kubectl get po -l app=sidecardemo --output jsonpath='{.items[0].metadata.name}')

# get the logs of the logshipper container within the sidecardemo Pod
kubectl logs $POD_NAME -c logshipper
```

### Utilize persistent and ephemeral volumes


**To see the storage classes that are available in your cluster.**

```bash
kubectl get storageclasses
```
* The `azurefile-csi` storage class is used to create Azure File Shares and the `managed-csi` storage class is used to create Azure Managed Disks.
* The classes with `csi` in the name infer that the Container Storage Interface drivers are used to provision the underlying storage

### Implementing persistent storage with Azure Storage

**Run the following command to create the PVC.**

```bash
kubectl apply -f - <<EOF
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: pvc-azurefile
spec:
  accessModes:
    - ReadWriteMany
  resources:
    requests:
      storage: 1Gi
  storageClassName: azurefile-csi
EOF
```
* This will request a 1Gi persistent volume from the azurefile-csi storage class.
* The `accessModes` field specifies that the volume can be mounted as read-write by as many nodes as needed which is useful for shared volumes.

**The following command will delete the previous Deployment.**

```bash
kubectl delete deployment sidecardemo
```

**Create a new Deployment that uses the PVC to store the logs.**

```bash
kubectl apply -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: sidecardemo
spec:
  replicas: 1
  selector:
    matchLabels:
      app: sidecardemo
  template:
    metadata:
      labels:
        app: sidecardemo
    spec:
      containers:
        - name: mylogger
          image: alpine:latest
          command: ['sh', '-c', 'while true; do date >> /opt/logs.txt; sleep 1; done']
          volumeMounts:
            - name: data
              mountPath: /opt
      initContainers:
        - name: logshipper
          image: alpine:latest
          restartPolicy: Always
          command: ['sh', '-c', 'touch /opt/logs.txt && tail -F /opt/logs.txt']
          volumeMounts:
            - name: data
              mountPath: /opt
      volumes:
        - name: data
          persistentVolumeClaim:
            claimName: pvc-azurefile       # <-- use the pvc
EOF
```
**Run the following command to see the status of the PVC.**

```bash
kubectl get pvc pvc-azurefile
```

* You should see the PVC with a `STATUS` of `Bound`. This means the PVC has been successfully created and is ready to be used by the pod.

**To find the Node Resource Group by running the following command.**

```bash
az resource list \
--resource-group $AKS_NRG_NAME \
--query "[?contains(type, 'Microsoft.Storage/storageAccounts')]" | jq
```

**Command to get the name of the storage account of the File share.**
```bash
STORAGE_ACCOUNT_NAME=$(az resource list \

```

**Command to get the name of the file share.**

```bash
FILE_SHARE_NAME=$(kubectl get pvc pvc-azurefile -o jsonpath='{.spec.volumeName}')
```
**Command to list the files in the Azure File Share.**

```bash
az storage file list \
--account-name $STORAGE_ACCOUNT_NAME \
--share-name $FILE_SHARE_NAME | jq
```
* You should see a file with the name `logs.txt`.
* Additionally, you can use the Azure Portal to view the file share and download the `logs.txt` file to view its contents.
