### Understand multi-container Pod design patterns (e.g. sidecar, init and others)

* A Pod can contain one or more containers that share the same network namespace and storage volumes.
* You can think of a Pod as a small VM where apps can communicate with each other and share data.

### Sidecar containers

* The design pattern of running multiple containers in a Pod, is often referred to as the "sidecar pattern".
* This is useful when you have a main application container with one or more containers that provide additional functionality such as logging, monitoring, caching, networking, etc.

**The following command to create a simple application that writes logs to a file and a sidecar container that reads the logs from the file and outputs them to stdout.**

```bash
kubectl apply -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: sidecardemo
spec:
  replicas: 1
  selector:
    matchLabels:
      app: sidecardemo
  template:
    metadata:
      labels:
        app: sidecardemo
    spec:
      containers:
        - name: mylogger
          image: alpine:latest
          command: ['sh', '-c', 'while true; do date >> /opt/logs.txt; sleep 1; done']
          volumeMounts:
            - name: data
              mountPath: /opt
      initContainers:
        - name: logshipper
          image: alpine:latest
          restartPolicy: Always
          command: ['sh', '-c', 'touch /opt/logs.txt && tail -F /opt/logs.txt']
          volumeMounts:
            - name: data
              mountPath: /opt
      volumes:
        - name: data
          emptyDir: {}
EOF
```

* Typically, we use YAML manifest files with `kubectl apply -f` command.
* Here we are using the `kubectl apply -f -` command to create the resources directly.

**This will create a Pod with two containers: `mylogger` and `logshipper`.**

* **mylogger** writes the current date to a file every second
* **logshipper** reads the file and outputs the contents to stdout

Run the following command to see the logs of the logshipper container.

```bash
# get the name of the sidecardemo Pod
POD_NAME=$(kubectl get po -l app=sidecardemo --output jsonpath='{.items[0].metadata.name}')

# get the logs of the logshipper container within the sidecardemo Pod
kubectl logs $POD_NAME -c logshipper
```

### Implementing persistent storage with Azure Storage

**Run the following command to create the PVC.**

```bash
kubectl apply -f - <<EOF
apiVersion: v1
```

### Application Deployment

* In this section, you will explore common deployment patterns (such as blue/green and canary), learn how rolling updates protect availability, and use declarative tools like Helm and Kustomize to package and configure your workloads.


**Blue/green deployment is a deployment strategy that allows you to deploy a new version of an application without downtime.**
**The idea is to have two identical environments, one for the current version (blue) and one for the new version (green).**


* In Kubernetes we create two separate Deployments for the blue and green environments and use a Service resource to route traffic to the correct Deployment based on the label of the Pods.
* Labels are key/value pairs that are attached to Kubernetes resources.
* They are used to identify and organize resources in the cluster.


- The Deployments will use a `ConfigMap` to store the HTML file that is served by the Nginx server.

In a Deployment spec, labels are not decorative; they are **selectors, grouping metadata, and automation anchors**.


---

### 1. Where Labels Appear in Your Deployment

You have labels in **two places**:

**A. Deployment metadata labels**

```yaml
metadata:
  name: nginx-blue
  labels:
    app: nginx-blue
```

**B. Pod template labels**

```yaml
template:
  metadata:
    labels:
      app: nginx-blue
```

**C. Selector labels**

```yaml
selector:
  matchLabels:
    app: nginx-blue
```

These three label contexts serve different functions.

---

### 2. Deployment Metadata Labels — Organizational Labels

### Location:

```yaml
metadata.labels
```

### Purpose:

These label the **Deployment object itself**, not the pods.

Typical uses:

* Resource organization
* Monitoring queries
* CI/CD automation
* Cost allocation tagging
* Observability filtering

### Example operational queries:

```bash
kubectl get deployments -l app=nginx-blue
```

or:

```
Prometheus scraping rules
Azure/GCP tagging
GitOps targeting
```

These labels do **NOT control pod selection**.

They’re purely descriptive.

---

# 3. Selector Labels — Control Plane Binding

### Location:

```yaml
spec.selector.matchLabels
```

This is critical.

It tells Kubernetes:

> "Which pods belong to this Deployment?"

This drives:

* Replica management
* Rolling updates
* Pod lifecycle control
* Ownership relationships

---

### Important rule:

**Selector labels must match pod template labels exactly.**

Otherwise:

* Deployment won’t manage pods.
* Scaling breaks.
* Updates fail.

This is why you see duplication.

---

# 4. Pod Template Labels — Identity Labels

### Location:

```yaml
spec.template.metadata.labels
```

These labels are applied to:

Actual running pods.

They enable:

### A. Deployment ownership

Selector must match them.

### B. Service routing

Example:

```yaml
spec:
  selector:
    app: nginx-blue
```

in a Service will route traffic to those pods.

### C. Operational grouping

Examples:

```
app=nginx-blue
tier=frontend
env=prod
version=v2
```

---

# 5. Why Duplication Exists (Architectural Reason)

Kubernetes separates:

| Concept           | Role                      |
| ----------------- | ------------------------- |
| Deployment labels | Metadata about controller |
| Selector labels   | Pod management contract   |
| Pod labels        | Pod identity              |

This separation enables:

* Independent metadata tagging
* Safe rolling updates
* Flexible routing strategies
* Multi-controller coexistence.

**To expose the nginx-blue Pod using a Service resource and make sure to set the selector to match the Pod label of the nginx-blue Pod.**

```bash
kubectl expose deploy/nginx-blue --selector app=nginx-blue --name mynginx --type LoadBalancer
```

* The `kubectl expose` command creates a Service resource that acts as a load balancer for the nginx-blue Pod.
* The `--selector` flag specifies the label selector to use to match the Pod, and the `--name` flag specifies the name of the Service resource.
* The `--type LoadBalancer` flag specifies that the Service resource should be of type LoadBalancer, which will create an external Azure Load Balancer to route external traffic to the Service.

### Canary Regions

* It is a deployment strategy that allows you to deploy a new version of an application to a small subset of users before rolling it out to the entire user base.
* The service will route traffic to any Pod that matches the label but you can control the percentage of traffic that is routed to each Deployment by scaling the replicas of each Deployment.

---
**To update the Deployments to add a new label to the Pods.**

**Do not perform this in Prod**


```bash
kubectl patch deploy nginx-blue --type='json' -p='[{"op": "add", "path": "/spec/template/metadata/labels/project", "value": "mynginx"}]'
kubectl patch deploy nginx-green --type='json' -p='[{"op": "add", "path": "/spec/template/metadata/labels/project", "value": "mynginx"}]'
```

- not recommended for production use.
- The `kubectl patch` command is used to update the existing resource.
- It is effectively like an update command but it allows you to update only specific fields of the resource.
- In this case, we are using the `--type='json'` flag to specify that we are using a JSON patch to update the resource.
- The `-p` flag specifies the patch to apply to the resource and the `op` field specifies the operation to perform.
- In this case, we are using the `add` operation to add a new label to the Pod template metadata.
---

**To remove the old label selector and add the new label selector to the Service resource.**

```bash
kubectl patch service mynginx --type='json' -p='[{"op": "remove", "path": "/spec/selector/app"}, {"op": "add", "path": "/spec/selector/project", "value": "mynginx"}]'
```

**To scale the nginx-blue Deployment to 1 replica and the nginx-green Deployment to 3 replicas.**

```bash
kubectl scale deploy nginx-blue --replicas=1
kubectl scale deploy nginx-green --replicas=3
```

**Now that you have the nginx-blue and nginx-green Deployments running with the same label, you can test the canary deployment strategy by running a curl command to see the response.**

```bash
while true; do curl -s "http://$(kubectl get svc mynginx --output jsonpath='{.status.loadBalancer.ingress[0].ip}')" | grep color:; done
```

- You should see the response alternating between blue and green text. This is because the Service resource is routing traffic to both Deployments based on the label selector.
- Press `Ctrl+C` to stop the curl command.
- Scale the nginx-blue Deployment to 0 replicas and the nginx-green Deployment to 4 replicas.

```bash
kubectl scale deploy nginx-blue --replicas=0
```
- Now if you run the curl command again, you should only see Nginx welcome page content with green text.
