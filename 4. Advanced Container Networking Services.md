### Advanced Container Networking Services (ACNS) enhances AKS operational capabilities through two key pillars:

### AKS Network & Observability Stack

The components below are ordered to follow the data journey: the **networking foundation** creates and secures pod traffic, the **observability engine** watches that traffic, it produces **two different kinds of data** (detailed logs vs. summarized metrics), and finally those are sent to **storage and visualization backends**.

| Layer | Component | Improved Definition (AKS Context) |
| :---- | :-------- | :-------------------------------- |
| **1. Networking foundation** | **CNI (Container Network Interface)** | The Kubernetes networking framework responsible for assigning IP addresses to pods and connecting them to the network. In AKS, common options include **Azure CNI Overlay** (pods use an internal overlay network) and **Azure CNI Flat** (pods receive IPs directly from the Azure VNet). |
| **1. Networking foundation** | **Cilium** | An **eBPF-based CNI and security platform** that replaces traditional iptables rules with kernel-level packet processing. It provides networking, network policies, load balancing, service discovery, and observability with lower overhead and better scalability. |
| **1. Networking foundation** | **ACNS (Advanced Container Networking Services)** | Azure's managed integration of Cilium that adds enterprise features such as **network policies, flow logs, network metrics, Hubble observability, and FQDN filtering**, all managed through native AKS and Azure tooling. |
| **2. Observability engine** | **Hubble** | Cilium's observability layer that uses eBPF events to **watch traffic in real time** (pod-to-pod, pod-to-service, and external communications, including DNS and HTTP). Hubble is the *source* that generates both the flow logs and the metrics described below. |
| **3a. Data output — detailed records** | **Container Network Flow Logs** | The **"itemized phone bill."** Records **one entry per individual connection** — Source IP, destination IP, ports, protocol (TCP/UDP/ICMP), bytes, packets, and whether the traffic was *allowed or dropped*. Used for troubleshooting, auditing, and forensic investigation. **Answers: "Who talked to whom, and was it allowed?"** Stored as text logs in **Log Analytics** (queried with KQL). |
| **3b. Data output — summarized numbers** | **Container Network Metrics** | The **"monthly usage graph."** **Aggregates** many connections into numeric **time-series** values — total throughput, connection counts, packet-drop rates, and DNS/HTTP latency (with L7 visibility). Used for dashboards, trends, and alerting. **Answers: "How much traffic, how many drops, what latency — over time?"** Stored in **Prometheus**. |
| **4. Storage & visualization** | **Prometheus** | The time-series database that **scrapes and stores the metrics** from Kubernetes components, applications, and ACNS/Cilium. In AKS, commonly provided through **Azure Managed Prometheus**. |
| **4. Storage & visualization** | **Grafana** | The visualization platform that turns Prometheus metrics into **graphs, heatmaps, service maps, and alerts**. AKS commonly integrates with **Azure Managed Grafana**. |
| **4. Storage & visualization** | **Log Analytics Workspace** | Azure's centralized **log** store that retains **textual and event-based data** — container logs, flow logs, diagnostics, and audit records — queried using **Kusto Query Language (KQL)**. |

> [!TIP]
> **Flow Logs vs. Metrics — the key difference:** Flow Logs are **individual records** (one row per connection, kept as *logs* in Log Analytics), answering *"who talked to whom?"*. Metrics are **aggregated numbers over time** (counters and rates, kept in *Prometheus*), answering *"how much / how many / how fast?"*. Same underlying traffic, two different views: **detailed evidence** vs. **summarized trends**.

### Diffrence between Azure Log Analytics and Prometheus

* **Azure Monitor / Log Analytics** primarily collects **platform metrics and logs**.
* **Prometheus** primarily collects **application metrics and Kubernetes metrics**.

However, there is some overlap.

| Type of Metric                                           | Azure Monitor / Log Analytics | Prometheus       |
| -------------------------------------------------------- | ----------------------------- | ---------------- |
| Azure platform metrics (VM CPU, Disk, Network)           | ✅ Primary source              | ❌ Usually not    |
| Kubernetes node metrics                                  | ✅ Yes                         | ✅ Yes            |
| Kubernetes component metrics (API server, kubelet, etc.) | Limited                       | ✅ Primary source |
| Application custom metrics (`/metrics`)                  | Limited                       | ✅ Primary source |
| Business metrics (orders, payments, users)               | ❌ Not automatically           | ✅ Yes            |
| Logs (application, container, audit)                     | ✅ Yes                         | ❌ No             |

### An easy way to remember

* **Azure Monitor / Log Analytics** → **Infrastructure & platform observability** (plus logs)
* **Prometheus** → **Application & Kubernetes observability** (metrics)

Here is the breakdown:

| Part | What it is | What it does |
| :--- | :--------- | :----------- |
| `kubectl exec` | kubectl subcommand | Executes a command **inside** a running pod. |
| `-n pets` | Flag (namespace) | Looks in the `pets` namespace. |
| `-it` | Flags (`-i` + `-t`) | `-i` keeps **stdin** open; `-t` allocates a **terminal (TTY)** for interactive I/O. |
| `$( ... )` | Command substitution | Runs the inner command first and substitutes its output (the pod name) into the outer command. |
| `kubectl get po -n pets -l app=order-service` | Inner command | Lists pods in `pets` labeled `app=order-service`. |
| `-ojsonpath='{.items[0].metadata.name}'` | Output format | Extracts just the **name of the first matching pod** from the JSON result. |
| `-c order-service` | Flag (container) | Selects the `order-service` container (a pod can have multiple containers). |
| `--` | Separator | Marks the end of kubectl's own flags; everything after runs **inside the container**. |
| `sh -c '...'` | Shell wrapper | Starts a shell in the container and runs the quoted command. |
| `wget www.bing.com` | In-container command | Attempts to fetch the page from `www.bing.com`. |
| `--spider` | wget flag | **Don't download** — just check whether the URL is reachable. |

### Configuring FQDN Filtering


#### Create an FQDN Policy

**This policy limits egress and permits access only to specified URLs, ensuring controlled outbound traffic.**




> [!NOTE]
> For FQDN filtering to work in Cilium, you need both components working together:
> 1. DNS rules with matchPattern - Allow the DNS query to resolve the domain name
> 2. toFQDNs rules - Allow the connection to the resolved IP address
> 
> Suppose if api.github.com is only in the `toFQDNs list` but not in the `DNS matchPattern` rules it will fail because the DNS query gets blocked first, so the pod never attempts the actual connection.
> In contrast, `*.microsoft.com` will work if it's in both sections.

The extra entries under rules.dns are not for additional destinations; they are there to accommodate the various DNS queries that the resolver might generate while trying to resolve *.microsoft.com.

When a pod inside Kubernetes wants to talk to an external site like microsoft.com, it doesn't just look up microsoft.com right away. Because of how Kubernetes networking is built, the pod automatically plays a guessing game by appending its own internal cluster names to the end of the request first.  

The pod will ask the DNS server for these names in order:  

1. "Hey, is there a microsoft.com.pets.svc.cluster.local inside our namespace?"  

2. "No? Okay, how about microsoft.com.svc.cluster.local?"  

3. "Still no? What about microsoft.com.cluster.local?"

4. "What about our cloud provider's internal network? microsoft.com...internal.cloudapp.net?"

5. "Fine, let's just look up the real, bare domain: microsoft.com."

Because Cilium is a Zero-Trust network security tool, it will block any DNS query that isn't explicitly allowed. If you only allowed *.microsoft.com, the pod's first 4 "guessed" queries would be blocked, causing the DNS lookup to fail before it ever reached the final, correct name.

Note: You also have rabbitmq.pets.svc.cluster.local here because your order-service needs to talk to your internal RabbitMQ database, which requires its own DNS lookup!

```yaml
rabbitmq.pets.svc.cluster.local
*.microsoft.com
*.microsoft.com.cluster.local
*.microsoft.com.pets.svc.cluster.local
*.microsoft.com.*.*.internal.cloudapp.net
*.microsoft.com.svc.cluster.local
```

Great question — this trips up almost everyone. The short answer:

> The two sections control **two different phases**, and they see **different names**.
> - `rules: dns:` filters the **question the pod asks** (DNS lookup) → the pod asks for *many* name variants.
> - `toFQDNs:` filters the **place the pod connects to** (the real resolved IP) → only *one* real name matters.

Let me unpack it.

## The core problem: the pod doesn't actually ask for `www.microsoft.com`

Because of how Kubernetes DNS is configured, a pod's `/etc/resolv.conf` looks roughly like this:

```
search pets.svc.cluster.local svc.cluster.local cluster.local <something>.internal.cloudapp.net
options ndots:5
```

That `ndots:5` rule means: **"If the name has fewer than 5 dots, try appending each search domain *first* before trying the bare name."**

`www.microsoft.com` has only 2 dots → so the resolver does **not** ask for `www.microsoft.com` first. It generates this sequence of **real, separate DNS queries**:

| # | The name actually sent to DNS | Does `*.microsoft.com` match it? |
| :- | :--- | :--- |
| 1 | `www.microsoft.com.pets.svc.cluster.local` | ❌ No |
| 2 | `www.microsoft.com.svc.cluster.local` | ❌ No |
| 3 | `www.microsoft.com.cluster.local` | ❌ No |
| 4 | `www.microsoft.com.<...>.internal.cloudapp.net` | ❌ No |
| 5 | `www.microsoft.com` (finally the bare name) | ✅ Yes |

## Why `*.microsoft.com` alone is not enough in the `dns:` section

The pattern `*.microsoft.com` only matches names that **end in** `.microsoft.com`. But queries #1–#4 **end in** `.cluster.local` or `.cloudapp.net` — so they do **not** match.

Cilium is **zero-trust**: any DNS query that doesn't match an allowed pattern gets **blocked/dropped**. So if you only listed `*.microsoft.com`:
- Queries #1–#4 get **dropped** → the resolver waits, times out, retries → **slow failures and errors** before it ever reaches query #5.

That's why the extra `matchPattern` entries exist — they simply say *"let those 'guessed' intermediate lookups through too, so resolution can finish cleanly and fast."* They are **not** extra destinations; they're just the noisy variants the resolver is guaranteed to generate.

## Why `toFQDNs:` only needs `*.microsoft.com`

`toFQDNs` controls the **actual connection**, which only happens **after** a name resolves to a **real IP address**.

- Queries #1–#4 return **NXDOMAIN** (they don't exist) → there's nothing to connect to.
- Only query #5, `www.microsoft.com`, returns a **real IP**.

So the only destination the pod ever truly connects to is a `*.microsoft.com` address → one pattern covers it.

## Summary

| Section | Governs | What it filters | Needs the extra variants? |
| :--- | :--- | :--- | :--- |
| `rules: dns:` | The **lookup** phase | Every **name the resolver asks** (many, due to search domains) | ✅ **Yes** — must allow all guessed variants |
| `toFQDNs:` | The **connect** phase | The **real resolved IP** of the final domain | ❌ **No** — only the real domain resolves |

## Can you actually simplify it? (Yes)

If you want to avoid the long list, force the resolver to skip the search-domain guessing:

1. **Use a fully-qualified name with a trailing dot** in the app — e.g. request `www.microsoft.com.` (note the final `.`). The trailing dot marks it absolute, so no search domains are appended → only query #5 happens, and `*.microsoft.com` alone works in both sections.

2. **Lower `ndots` on the pod** via `dnsConfig` so the bare name is tried first:
   ```yaml
   spec:
     dnsConfig:
       options:
         - name: ndots
           value: "1"
   ```
   With `ndots:1`, `www.microsoft.com` (2 dots ≥ 1) is tried **as-is first**, skipping the cluster-suffix guesses.

Either approach removes the need for the extra `matchPattern` lines — the list in the lab exists because it works with the **default** Kubernetes DNS behavior without modifying the app or pod spec.

Having the extra `matchPattern` entries helps in **three concrete ways** — all about letting the resolver's "guessed" lookups (the search-domain variants) pass through instead of being blocked by Cilium.

## 1. It prevents DNS resolution from failing

Cilium is zero-trust: **any** DNS query that doesn't match an allowed pattern is **dropped**. The resolver asks for the variants in order:

```
1. www.microsoft.com.pets.svc.cluster.local   ← if not allowed → DROPPED
2. www.microsoft.com.svc.cluster.local        ← if not allowed → DROPPED
3. www.microsoft.com.cluster.local            ← if not allowed → DROPPED
4. www.microsoft.com.<...>.internal.cloudapp.net ← if not allowed → DROPPED
5. www.microsoft.com                          ← allowed → resolves ✅
```

With **only 1 entry** (`*.microsoft.com`), queries 1–4 are dropped. Depending on the resolver, a dropped (unanswered) query can make it **give up or error out** before it ever reaches query 5 — so the lookup **fails even though the domain is allowed**.

With **all entries present**, queries 1–4 are allowed through, return a normal "does not exist" (NXDOMAIN), and the resolver cleanly moves on to query 5.

## 2. It makes resolution fast instead of slow

A **dropped** query gets **no response at all**, so the resolver has to **wait for a timeout** (often 2–5 seconds each) and retry before trying the next variant.

| Setup | What happens to variants 1–4 | Time cost |
| :--- | :--- | :--- |
| **1 entry only** | Dropped → no reply → resolver waits + retries | **Seconds of delay** (can cause app timeouts) |
| **All entries** | Allowed → instant NXDOMAIN reply | **Milliseconds** — resolver skips straight to #5 |

So the extra entries turn a slow, timeout-ridden lookup into a fast one.

## 3. It keeps logs and troubleshooting clean

Every dropped DNS query shows up as a **`policy_denied` drop** in Hubble/flow logs. With only 1 entry, every single external lookup produces **4 noisy "denied" records** for traffic that was actually supposed to succeed — making real problems hard to spot. Allowing the variants means only genuinely unwanted lookups appear as denials.

## The key point

The extra entries **don't grant access to anything new**. Queries 1–4 return NXDOMAIN and connect to nothing (that's why `toFQDNs` still needs only `*.microsoft.com`). They exist purely to let the resolver's *unavoidable* guessing finish **successfully, quickly, and quietly** instead of being blocked, slow, and noisy.

> **Analogy:** You want to reach one office (`www.microsoft.com`), but the building's rule forces you to knock on 4 wrong doors first before the right one. Listing all 5 doors doesn't give you access to extra rooms — it just stops a guard from tackling you at each wrong door so you can actually get to the right one.

This is a **KQL (Kusto Query Language)** query — the language you use to search logs in Azure Log Analytics. Think of it like asking questions about a giant spreadsheet of network events. Data flows **top to bottom**, and each `|` (pipe) passes the result to the next step, refining it further.

## The big picture

> *"In the last 30 minutes, find network connections involving the `pets` app that were **blocked**, then show me the top 20 pod pairs with the most blocks."*

## Line-by-line

| Line | Plain-English meaning |
| :--- | :--- |
| `ContainerNetworkLogs` | **The table (the "spreadsheet")** we're searching — every network flow record collected from your cluster. |
| `\| where TimeGenerated > ago(30m)` | **Keep only recent rows** — events from the **last 30 minutes**. (`ago(30m)` = "30 minutes ago".) |
| `\| where SourceNamespace == "pets" or DestinationNamespace == "pets"` | **Keep only rows involving the `pets` app** — where the traffic either **came from** or **went to** the `pets` namespace. |
| `\| where Verdict == "DROPPED"` | **Keep only blocked traffic** — connections that were **denied** (e.g., by a network policy), not allowed ones. |
| `\| summarize DroppedFlows = count() by TrafficDirection, SourcePodName, DestinationPodName` | **Group and count.** Bundle rows that share the same *direction + source pod + destination pod*, and count how many drops each group had. Call that count `DroppedFlows`. |
| `\| order by DroppedFlows desc` | **Sort** so the pods with the **most drops appear first** (`desc` = highest → lowest). |
| `\| take 20` | **Show only the top 20** results. |

## The two key concepts

1. **`where` = filter** (throw away rows you don't want). This query filters three times: by *time*, by *namespace*, then by *verdict*.

2. **`summarize ... by ...` = group + aggregate** (like a pivot table). It collapses many individual drop events into a tidy count **per pod-pair**, so instead of thousands of raw rows you get something like:

   | TrafficDirection | SourcePodName | DestinationPodName | DroppedFlows |
   | :--- | :--- | :--- | ---: |
   | egress | store-front-abc | order-service-xyz | 143 |
   | egress | order-service-xyz | product-service-def | 87 |
   | … | … | … | … |

## Why you'd run this

It answers: **"Which pods are being blocked the most right now?"** — the fastest way to spot a misconfigured network policy or a service that can't reach something it needs. The pod pair at the top of the list is where you'd start troubleshooting.

**Query 3:** *Diagnose EGRESS - Separate DNS Failures from Connection Failures*

> DNS queries blocked (api.github.com, google.com, bing.com)
> Pattern identified: DNS is being selectively filtered

**Key insight:** Some domains fail at the DNS level - they never even get to attempt the HTTPS connection. This suggests the DNS `matchPattern` rules are too restrictive.

**Query 4:** *Complete Traffic Pattern - DNS + HTTPS Correlation*

* Now let's correlate DNS queries with HTTPS connection attempts to understand the full flow:

```
ContainerNetworkLogs
| where TimeGenerated > ago(30m)
| where SourceNamespace == "pets"
| where SourcePodName contains "store-front"
| where TrafficDirection == "EGRESS"
| extend SrcIP = tostring(IP.source), DstIP = tostring(IP.destination)
| extend Layer4Data = parse_json(Layer4)
| extend DstPort = coalesce(tostring(Layer4Data.TCP.destination_port), tostring(Layer4Data.UDP.destination_port))
| where DstPort in ("53", "443")  // DNS and HTTPS
| where isnotempty(DstIP)
| project TimeGenerated, DstIP, DstPort, Verdict
| order by TimeGenerated asc
| take 50
```

**What you'll discover:**


*Cumulative insights:*

**For microsoft.com (✅ Working):**

* Step 1: DNS query to 10.0.0.10 port 53 → FORWARDED
* Step 2: HTTPS connection to 23.192.18.101 port 443 → FORWARDED
* Result: Network allows traffic (HTTP 403 from server is expected)

**For api.github.com, google.com, bing.com (❌ Failing):**

* Step 1: DNS query to 10.0.0.10 port 53 → DROPPED
* Step 2: HTTPS connection → Never attempted (DNS failed first)
* Result: "bad address" error

*Root cause confirmed:* The FQDN policy is missing these domains from the DNS matchPattern rules. Even though api.github.com is in the toFQDNs list, the DNS query gets blocked first.

*Educational insight:* This demonstrates a common FQDN policy misconfiguration - you need BOTH DNS rules AND toFQDNs rules for external access to work.

**Query 6:** *Complete Diagnosis*

* Get a comprehensive view of all traffic patterns to confirm your diagnosis:

```
ContainerNetworkLogs
| where TimeGenerated > ago(30m)
| where SourceNamespace == "pets" or DestinationNamespace == "pets"
| summarize
    TotalFlows = count(),
    DroppedFlows = countif(Verdict == "DROPPED"),
    ForwardedFlows = countif(Verdict == "FORWARDED")
    by TrafficDirection, SourcePodName, DestinationPodName
| extend DropRate = round((DroppedFlows * 100.0) / TotalFlows, 2)
| where TotalFlows > 5  // Filter out noise
| order by DroppedFlows desc
| take 20
```

**What you'll discover:**

**Problem 1**: External Users Can't Access the Application

* *Evidence:* INGRESS to store-front has 100% drop rate
* *Root Cause:* Chaos policy has ingress: - fromEndpoints: [] (allow from nowhere = block all)
* *Fix Needed:* Restore original policy with fromEntities: - world

**Problem 2:** External API Calls Failing

* *Evidence:* EGRESS to external has 25.8% drop rate, DNS queries have 26.7% drop rate
* *Root Cause:* FQDN policy missing domains (api.github.com, google.com, bing.com) from DNS matchPattern rules
* *Fix Needed:* Add missing domains to DNS patterns OR remove from toFQDNs list

**Problem 3:** Internal Communication Working Fine

* *Evidence:* EGRESS between pods in pets namespace has greater than 5% drop rate (filtered out)
* *Conclusion:* Application code and internal services are healthy

### Optimize Metrics Collection with Container Network Metrics Filtering

* Instead of collecting everything and filtering later (expensive).
* We define what matters at the source (cost-effective).


### How metrics filtering works

* Metrics filtering uses a Kubernetes **Custom Resource Definition (CRD)** called `ContainerNetworkMetric`.
* You define filtering rules that tell the Cilium agents which Hubble metrics to collect and which to skip.

**Key characteristics:**

* **One CRD per cluster** — only a single `ContainerNetworkMetric` resource can exist
* **Dynamic reconciliation** — changes take effect in approximately 30 seconds without restarting Cilium agents or Prometheus
* **No CRD = collect everything** — if you don't apply a CRD, all metrics are collected (the current default behavior)
* **Four metric types** — `dns`, `flow`, `tcp`, and `drop`
* **Include and exclude filters** — target by namespace/pod, label selector, IP/CIDR, protocol, or verdict

### Stop the port-forward:

```bash
kill %1 2>/dev/null
```
Updated todo list

This is a **Bash command that stops a background job** and hides any error message. Breakdown:

| Part | Meaning |
| :--- | :--- |
| `kill` | Sends a signal to a process to **terminate** it (by default, the `TERM` / "please stop" signal). |
| `%1` | **Job number 1** — refers to the *first background job* in your current shell (not a process ID). The `%` prefix means "this is a job spec, not a PID." |
| `2>` | **Redirect file descriptor 2**, which is **stderr** (the error-message stream). |
| `/dev/null` | The system **"trash can"** — anything sent here is silently discarded. |



## Quick contrast

- `kill %1` → stop job #1 (**errors shown** if it's not running).
- `kill %1 2>/dev/null` → stop job #1 (**errors hidden** — clean cleanup).
- `kill 1234` → stop the process with **PID 1234** (a real process ID, no `%`).

> **In short:** it's a **tidy cleanup command** — "shut down the background task I started earlier, and don't nag me if it's already gone."

### Apply a targeted metrics filter for the pets namespace

* apply a filter that focuses metrics collection on the `pets` namespace (where your application workloads run) and excludes noisy system namespaces.

```yaml
kubectl apply -f - <<'EOF'
apiVersion: acn.azure.com/v1alpha1
kind: ContainerNetworkMetric
metadata:
  name: container-network-metric
spec:
  filters:
    # Collect flow metrics only for the pets namespace
    - metric: flow
      includeFilters:
        - name: pets-app-flows
          from:
            labelSelector:
              matchExpressions:
                - key: k8s.io/namespace
                  operator: In
                  values:
                    - pets
        - name: pets-app-flows-ingress
          to:
            labelSelector:
              matchExpressions:
                - key: k8s.io/namespace
                  operator: In
                  values:
                    - pets

    # Collect DNS metrics but exclude kube-system noise
    - metric: dns
      excludeFilters:
        - name: exclude-system-dns
          from:
            namespacedPod:
              - "kube-system/"

    # Collect TCP metrics only for the pets namespace
    - metric: tcp
      includeFilters:
        - name: pets-tcp-metrics
          from:
            labelSelector:
              matchExpressions:
                - key: k8s.io/namespace
                  operator: In
                  values:
                    - pets

    # Collect all drop metrics (always useful for troubleshooting)
    - metric: drop
      excludeFilters:
        - name: exclude-system-drops
          from:
            namespacedPod:
              - "kube-system/"
EOF
```

### Verify the CRD was applied:

```bash
kubectl get ContainerNetworkMetric
```
You should see your `container-network-metric` resource listed in 30 seconds.

**Understanding the filter**

| Metric type | Filter strategy | What it collects | What it skips |
|---|---|---|---|
| `flow` | Include only `pets` namespace | Pod-to-pod flows for store-front, order-service, product-service, etc. | All kube-system, monitoring, and other namespace flows |
| `dns` | Exclude `kube-system` | DNS queries from `pets` and other app namespaces | CoreDNS internal metrics, system DNS chatter |
| `tcp` | Include only `pets` namespace | TCP connection states for your application | System-level TCP connections |
| `drop` | Exclude `kube-system` | Dropped packets in application namespaces (useful for policy debugging) | Expected system-level drops |

### Experiment with different filter strategies

**Strategy A: Monitor only dropped packets (minimum cost)**

```yaml
kubectl apply -f - <<'EOF'
apiVersion: acn.azure.com/v1alpha1
kind: ContainerNetworkMetric
metadata:
  name: container-network-metric
spec:
  filters:
    - metric: drop
      includeFilters:
        - name: all-drops
          verdict:
            - dropped
EOF
```

* This is the most aggressive cost reduction — you only collect metrics for dropped packets, which are the most actionable for troubleshooting.

---

**Strategy B: Production vs. non-production filtering**

```YAML
kubectl apply -f - <<'EOF'
apiVersion: acn.azure.com/v1alpha1
kind: ContainerNetworkMetric
metadata:
  name: container-network-metric
spec:
  filters:
    - metric: flow
      includeFilters:
        - name: production-only
          from:
            labelSelector:
              matchLabels:
                environment: production
    - metric: dns
      includeFilters:
        - name: production-dns
          from:
            labelSelector:
              matchLabels:
                environment: production
    - metric: tcp
      includeFilters:
        - name: production-tcp
          from:
            labelSelector:
              matchLabels:
                environment: production
    - metric: drop
      includeFilters:
        - name: all-env-drops
          verdict:
            - dropped

### Configure the client with Hubble certificate:

```bash
#!/usr/bin/env bash

set -euo pipefail
set -x

# Directory where certificates will be stored
CERT_DIR="$(pwd)/.certs"
mkdir -p "$CERT_DIR"

declare -A CERT_FILES=(
  ["tls.crt"]="tls-client-cert-file"
  ["tls.key"]="tls-client-key-file"
  ["ca.crt"]="tls-ca-cert-files"
)

for FILE in "${!CERT_FILES[@]}"; do
  KEY="${CERT_FILES[$FILE]}"
  JSONPATH="{.data['${FILE//./\\.}']}"

  # Retrieve the secret and decode it
  kubectl get secret hubble-relay-client-certs -n kube-system -o jsonpath="${JSONPATH}" | base64 -d > "$CERT_DIR/$FILE"

  # Set the appropriate hubble CLI config
  hubble config set "$KEY" "$CERT_DIR/$FILE"
done

hubble config set tls true
hubble config set tls-server-name instance.hubble-relay.cilium.io
```

### Visualize Traffic with Hubble UI

**Install Hubble UI**

* Run the following command to apply the Hubble UI manifest file from the assets folder.

```bash
curl -o acns-hubble-ui.yaml https://raw.githubusercontent.com/Azure-Samples/aks-labs/refs/heads/main/docs/networking/assets/acns-hubble-ui.yaml
```

* Optionally, run the following command to take a look at the Hubble UI manifest file.

```bash
cat acns-hubble-ui.yaml
```

* Apply the hubble-ui.yaml manifest to your cluster, using the following command:

```bash
kubectl apply -f acns-hubble-ui.yaml
```

**Forward Hubble Relay Traffic**

* Set up port forwarding for Hubble UI using the kubectl port-forward command.

```bash
kubectl -n kube-system port-forward svc/hubble-ui 12000:80
```

**Access Hubble UI**

* Access Hubble UI by entering `http://localhost:12000/` into your web browser.
