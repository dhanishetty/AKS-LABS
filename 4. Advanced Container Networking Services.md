# CNI vs Cilium vs ACNS

These three often get lumped together, but they actually sit at different layers of the stack — they're not really "either/or" competitors. Here's how they relate, from the ground up.

The quick mental model
**CNI** = the standard/interface (the "what")
**Cilium** = one implementation of that interface, using eBPF (the "how")
**ACNS** = a paid add-on suite that layers extra security & observability on top of Cilium (the "extras")


---

### 1. CNI (Container Networking Interface)

CNI is a generic Kubernetes specification that defines how pods get IP addresses and how traffic is routed. It's not Azure-specific. In AKS, the CNI plugin determines your IP address management (IPAM) model:

**Azure CNI – Overlay (recommended default)**: Pod IPs come from a private CIDR outside the VNet, so you avoid IP exhaustion. Scales to ~5,000 nodes / 250,000 pods.
**Azure CNI – Pod Subnet (Dynamic/Static)**: Pod IPs come from a delegated VNet subnet, so pods are directly routable from outside the cluster — but you need careful IP planning.
**Kubenet (legacy)**: Simple but limited (400-node route table limit); being phased out.

Think of CNI as addressing & routing model.

---

### 2. Cilium (the eBPF data plane)

Cilium is an open-source, eBPF-based data plane — a specific way of implementing networking. In AKS it shows up as **"Azure CNI Powered by Cilium"**, which fuses Azure's CNI control plane with Cilium's eBPF data plane.

Key benefits over the standard iptables-based data plane:

Replaces iptables with eBPF → faster forwarding, lower CPU
Built-in network policy enforcement — no need for a separate engine like Calico or Azure NPM
Better service routing & observability
Scales to larger clusters
Works with any IPAM mode (Overlay, Dynamic, or Static)

Think of Cilium as the high-performance engine under the CNI.

---

### 3. ACNS (Advanced Container Networking Services)

ACNS is a paid Azure add-on (`--enable-acns`) that bolts advanced capabilities onto your cluster. It's organized around a few pillars:

**Container Network Observability** — Hubble metrics + flow logs (DNS, pod-to-pod, service-to-service), integrated with Managed Prometheus & Grafana. ✅ Works on both Cilium and non-Cilium data planes.
**Container Network Security** — FQDN filtering + Layer 7 (HTTP/gRPC/Kafka) policies. ⚠️ Requires Azure CNI Powered by Cilium (K8s 1.29+).
**Container Network Performance** — eBPF host routing optimizations.
Encryption — WireGuard or Cilium mTLS for transparent pod-to-pod encryption (preview).

Important: ACNS is not free and the security/encryption features are gated behind Cilium. Observability is the "safe" entry point since it works everywhere.

Think of ACNS as the premium security + observability layer on top.

---

Putting it together
Layer	What it is	Required?	Key pointCNI	Networking interface/IPAM model	✅ Always (every cluster has one)	Choose Overlay vs Pod Subnet
Cilium	eBPF data plane implementation	Optional but recommended	--network-dataplane cilium
ACNS	Paid add-on (security + observability)	Optional	--enable-acns; security needs Cilium

A typical modern AKS command ties all three together:

az aks create \
  --network-plugin azure \
  --network-plugin-mode overlay \   # CNI (Overlay IPAM)
  --network-dataplane cilium \      # Cilium data plane
  --enable-acns                     # ACNS add-on


💡 Bottom line: They're not rivals — you stack them. You always have a CNI; you optionally choose Cilium as the data plane; and you optionally enable ACNS for advanced security/observability (with the security bits requiring Cilium).

# Detailed explanation of Hubble, Prometheus and Log Analytics

To understand how these three tools fit together in Azure Kubernetes Service (AKS), it helps to think of them as three distinct pillars of **observability**: network visibility (Hubble), performance metrics (Prometheus), and centralized data storage/analysis (Log Analytics Workspace).

Here is a breakdown of what each tool does and how they relate within AKS.

### 1. The Definitions

**Hubble (Network Observability)**

* **What it is:** Hubble is a fully distributed networking and security observability platform. It is built on top of Cilium (a popular Kubernetes networking solution) and uses a Linux kernel technology called eBPF.
* **What it does:** It acts as an incredibly advanced network microscope. It doesn't just see that "Pod A" is talking to "Pod B"; it sees the exact network flows, dropped packets, DNS resolution errors, and Layer 7 (HTTP/gRPC) traffic details, all with very low performance overhead.

**Prometheus (Metrics Monitoring)**

* **What it is:** An open-source systems monitoring and alerting toolkit that has become the industry standard for Kubernetes environments.
* **What it does:** Prometheus works by "scraping" (pulling) numerical metrics from your cluster. It tracks things like CPU usage, memory consumption, active HTTP requests, and custom application metrics (e.g., "how many items were added to the shopping cart"). It stores this data as time-series data.

**Log Analytics Workspace (Centralized Data Platform)**

* **What it is:** Log Analytics Workspace (LAW) is a core component of Azure Monitor. It is a highly scalable, centralized data repository in the Azure cloud.
* **What it does:** It acts as the giant bucket where all your logs, traces, and diagnostic data end up. Once the data is in the workspace, you use Kusto Query Language (KQL) to search, cross-reference, and analyze massive amounts of telemetry in seconds.

---

### 2. How They Are Related and Used in AKS

In a modern AKS deployment, these tools do not compete; they complement each other to give you a complete picture of cluster health and security. Microsoft Azure tightly integrates all three through **Azure Monitor**.

Here is how they work together in an AKS architecture:

#### The Collection Layer (Hubble & Prometheus)

When you deploy an application to AKS, you need to know if it is healthy.

* **Prometheus** (often deployed as *Azure Monitor managed service for Prometheus*) continuously scrapes your AKS nodes and pods to gather performance metrics. It tells you *if* a service is struggling (e.g., "CPU is at 99%").
* **Hubble** (often utilized under the hood when using Azure CNI powered by Cilium or Advanced Container Networking Services) monitors the actual network traffic between those pods. If Prometheus says a service is slow, Hubble can tell you *why* (e.g., "DNS lookups are timing out" or "A network policy is dropping packets between the frontend and the backend").

#### The Storage & Analysis Layer (Log Analytics Workspace)

Neither Hubble nor Prometheus are designed to store massive amounts of historical, searchable log data forever. That is where LAW comes in.

* Through a feature called **Container Insights**, AKS automatically forwards control plane logs, application stdout/stderr logs, and infrastructure telemetry directly into your Log Analytics Workspace.
* **The Integration:** When you enable advanced network observability features (like the Flow Logs mentioned in Advanced Container Networking labs), the telemetry captured by backend tools like Hubble/Retina is packaged and shipped to the **Log Analytics Workspace**.

#### The Visualization (The User Experience)

Because all this data ultimately feeds into the Azure ecosystem:

* You use **Grafana** (often Azure Managed Grafana) to visualize the metrics pulled by Prometheus.
* You use **Azure Workbooks** or KQL queries in the **Log Analytics Workspace** to visualize the network flow logs gathered by Hubble, alongside your standard application logs.

**In Summary:** Prometheus tells you the numerical health of your cluster, Hubble maps out and inspects the network traffic flowing through it, and Log Analytics Workspace is the central brain where you store, query, and analyze all that information together.

### Test Connectivity

This command is a **connectivity test** from inside your `order-service` pod to the internet.

```bash
kubectl exec -n pets -it $(kubectl get po -n pets -l app=order-service -ojsonpath='{.items[0].metadata.name}') -c order-service -- sh -c 'wget --spider www.bing.com'
```

expected Output = "Connecting to www.bing.com (23.39.174.152:80) remote file exists"

Here is the breakdown:

* `kubectl exec` = Execute a command inside the Pod
* `-n pets` = Look in the namespace pets
* `-it` = i is for "keep stdin open" and -t is for "allocate a terminal"
* `$(kubectl get po -n pets -l app=order-service -ojsonpath='{.items[0].metadata.name}')` = Gets the name of the first Pod matching
* `kubectl get po -n pets -l app=order-service` = pods with namespace pets and labels order-service.
* `-ojsonpath='{.items[0].metadata.name}'` = Extracts first name
* `-c order-service` = Select the specific container from different containers in a pod

* `--` = Separates Kubernetes arguments from the command that runs inside the container.
* `sh -c 'wget --spider www.bing.com'` = Runs inside the container
* `wget www.bing.com` = downloads the page.
* `--spider` = Don't download. Just check if the URL is reachable.

### In short:

It automatically grabs the name of your running **order-service** pod and tries to **connect to Bing.com** from inside it to verify that the pod has outgoing internet access.

* You should see output similar to the following:

```bash
Connecting to www.bing.com (13.107.21.237:80)
remote file exists
```

### Configuring FQDN Filtering


#### Create an FQDN Policy

**This policy limits egress and permits access only to specified URLs, ensuring controlled outbound traffic.**




> [!NOTE]
> For FQDN filtering to work in Cilium, you need both components working together:
> 1. DNS rules with matchPattern - Allow the DNS query to resolve the domain name
> 2. toFQDNs rules - Allow the connection to the resolved IP address
> 
> Suppose if api.github.com is only in the `toFQDNs list` but not in the `DNS matchPattern` rules it will fail because the DNS query gets blocked first, so the pod never attempts the actual connection.
> In contrast, `*.microsoft.com` will work if it's in both sections.

* Run the following command to download the FQDN policy manifest file.

```bash
curl -o acns-network-policy-fqdn.yaml https://raw.githubusercontent.com/Azure-Samples/aks-labs/refs/heads/main/docs/networking/assets/acns-network-policy-fqdn.yaml
```

* Take a look at the FQDN policy manifest file by running the following command.

```bash
cat acns-network-policy-fqdn.yaml
```
**Output**

```yaml
apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata:
  name: allow-order-service-traffic
spec:
  endpointSelector:
    matchLabels:
      app: order-service
  egress:
    - toEndpoints:
        - matchLabels:
            io.kubernetes.pod.namespace: kube-system
            k8s-app: kube-dns
        - matchLabels:
            io.kubernetes.pod.namespace: pets
            app: rabbitmq
      toPorts:
        - ports:
            - port: "53"
              protocol: UDP
          rules:
            dns:
              - matchPattern: "rabbitmq.pets.svc.cluster.local"
              - matchPattern: "*.microsoft.com"
              - matchPattern: "*.microsoft.com.cluster.local"
              - matchPattern: "*.microsoft.com.pets.svc.cluster.local"
              - matchPattern: "*.microsoft.com.*.*.internal.cloudapp.net"
              - matchPattern: "*.microsoft.com.svc.cluster.local"
    - toFQDNs:
        - matchPattern: "*.microsoft.com"
      toPorts:
        - ports:
            - port: "443"
              protocol: TCP
```

1. Only applies to Order Service Pods

```yaml
endpointSelector:
  matchLabels:
    app: order-service
```

2. Allows the order-service pods to send DNS requests to: `kube-dns (cluster DNS server)` & `rabbitmq pod` on port 53

```yaml
toEndpoints:
  - kube-dns
  - rabbitmq
toPorts:
  - port: "53"
    protocol: UDP
```

3. Restricts which DNS names can be resolved - Allowed DNS lookups include: 
   So the pod can only perform DNS lookups for these names.

The extra entries under rules.dns are not for additional destinations; they are there to accommodate the various DNS queries that the resolver might generate while trying to resolve *.microsoft.com.

When a pod inside Kubernetes wants to talk to an external site like microsoft.com, it doesn't just look up microsoft.com right away. Because of how Kubernetes networking is built, the pod automatically plays a guessing game by appending its own internal cluster names to the end of the request first.  

The pod will ask the DNS server for these names in order:  

1. "Hey, is there a microsoft.com.pets.svc.cluster.local inside our namespace?"  

2. "No? Okay, how about microsoft.com.svc.cluster.local?"  

3. "Still no? What about microsoft.com.cluster.local?"

4. "What about our cloud provider's internal network? microsoft.com...internal.cloudapp.net?"

5. "Fine, let's just look up the real, bare domain: microsoft.com."

Because Cilium is a Zero-Trust network security tool, it will block any DNS query that isn't explicitly allowed. If you only allowed *.microsoft.com, the pod's first 4 "guessed" queries would be blocked, causing the DNS lookup to fail before it ever reached the final, correct name.

Note: You also have rabbitmq.pets.svc.cluster.local here because your order-service needs to talk to your internal RabbitMQ database, which requires its own DNS lookup!

```yaml
rabbitmq.pets.svc.cluster.local
*.microsoft.com
*.microsoft.com.cluster.local
*.microsoft.com.pets.svc.cluster.local
*.microsoft.com.*.*.internal.cloudapp.net
*.microsoft.com.svc.cluster.local
```

4. Allows HTTPS traffic to Microsoft domains

```yaml
toFQDNs:
  - matchPattern: "*.microsoft.com"
```

Examples

```yaml
login.microsoft.com
api.microsoft.com
portal.microsoft.com
```

### Enable Flow Logs for the Pets Namespace

* To enable container network flow logs, you need to apply a `ContainerNetworkLog` custom resource that defines which network flows to capture.
* This filter helps to capture all traffic in the pets namespace.


* `<<`: This is the redirection operator that tells the shell to feed the subsequent text directly into the standard input (`stdin`) of the command.

* `-` (after `-f`): This flag tells `kubectl` to read the configuration from `stdin` instead of a physical file on your disk.

* `'EOF'`: This acts as the starting marker. By putting it in quotes, you prevent the shell from trying to expand any variables (like $) inside the YAML content.

* The content block: Everything between the first `EOF` and the final `EOF` is treated as the file content.

* The final `EOF`: This signals the end of the input block, telling the shell, "I am finished providing data; please process this now."

```yaml
kubectl apply -f - <<'EOF'
apiVersion: acn.azure.com/v1alpha1
kind: ContainerNetworkLog
metadata:
  name: testcnl # Cluster scoped
spec:
  includefilters: # List of filters
    - name: egress-filter # Capture egress traffic from pets namespace
      from:
        namespacedPod: # List of source namespace/pods. Prepend namespace with /
          - pets/order-service-
          - pets/product-service-
          - pets/rabbitmq
          - pets/store-front-
          - kube-system/core-dns-
      protocol: # List of protocols; can be tcp, udp, dns
        - tcp
        - udp
        - dns
      verdict: # List of verdicts; can be forwarded, dropped
        - forwarded
        - dropped

    - name: ingress-filter # Capture ingress traffic to pets namespace
      to:
        namespacedPod: # Destination pods
          - pets/store-front-
          - pets/order-service-
          - pets/product-service-
          - pets/rabbitmq
          - kube-system/core-dns-
      protocol:
        - tcp
        - udp
      verdict:
        - forwarded
        - dropped
EOF
```

**Verify the custom resource was created successfully:**

```bash
kubectl describe containernetworklog testcnl
```
You should see a `Status` field showing `State: CONFIGURED`. This means flow logs are now being collected for the pets namespace and sent to your Log Analytics workspace.

**Command to download the combined FQDN and L7 policy manifest file.**

```

**Output**
```
apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata:
  name: combined-fqdn-l7-policy
  namespace: pets
spec:
  endpointSelector:
    matchLabels:
      app: store-front
  
  # BLOCK ALL INGRESS TRAFFIC
  ingress: []  # Empty ingress = block all incoming traffic
  
  egress:
  # 1. Allow DNS to kube-dns (only for specific domains)
  - toEndpoints:
    - matchLabels:
        k8s:io.kubernetes.pod.namespace: kube-system
        k8s:k8s-app: kube-dns
    toPorts:
    - ports:
      - port: "53"
        protocol: ANY
      rules:
        dns:
          - matchPattern: "rabbitmq.pets.svc.cluster.local"
          - matchPattern: "*.microsoft.com"
          - matchPattern: "*.microsoft.com.cluster.local"
          - matchPattern: "*.microsoft.com.pets.svc.cluster.local"
          - matchPattern: "*.microsoft.com.*.*.internal.cloudapp.net"
          - matchPattern: "*.microsoft.com.svc.cluster.local"
          - matchPattern: "*.pets.svc.cluster.local"
          - matchPattern: "*.svc.cluster.local"
          # NOTE: api.github.com, google.com, and bing.com are NOT in DNS rules
          # This will cause DNS queries to fail for these domains

  # 2. Allow both HTTP and HTTPS to *.microsoft.com (FQDN filtering)
  - toFQDNs:
    - matchPattern: "*.microsoft.com"
    toPorts:
    - ports:
      - port: "80"
        protocol: TCP
      - port: "443"
        protocol: TCP

  # 3. Allow HTTPS to api.github.com (but DNS will fail - see note above)
  # This demonstrates a common misconfiguration: toFQDNs without DNS rules
  - toFQDNs:
    - matchName: api.github.com
    toPorts:
    - ports:
      - port: "443"
        protocol: TCP

  # 4. Allow internal backend communication with L7 HTTP rules
  - toEndpoints:
    - matchLabels:
        app: product-service
    toPorts:
    - ports:
      - port: "3002"
        protocol: TCP
      rules:
        http:
        - method: "GET"
          path: "/"
```

**Apply the combined FQDN and L7 policy to the pets namespace.**

### Diagnosis Using Flow Logs

**Query 1:** *What's Being Blocked?*

```

ContainerNetworkLogs
| where TimeGenerated > ago(30m)
| where SourceNamespace == "pets" or DestinationNamespace == "pets"
| where Verdict == "DROPPED"
| summarize 
    DroppedFlows = count()
    by TrafficDirection, SourcePodName, DestinationPodName
| order by DroppedFlows desc
| take 20
```
**What you'll discover:**
> If INGRESS to store-front has dropped flows - This confirms users can't access the application
> If EGRESS from store-front has dropped flows - Some external API calls are being blocked

---

**Query 2:** *Diagnose the INGRESS Problem - Who's Being Blocked?*

* let's see exactly which external connections are being dropped:


```
ContainerNetworkLogs
| where TimeGenerated > ago(30m)
| where DestinationNamespace == "pets"
| where DestinationPodName contains "store-front"
| where TrafficDirection == "INGRESS"
| where Verdict == "DROPPED"
| extend SrcIP = tostring(IP.source), DstIP = tostring(IP.destination)
| extend Layer4Data = parse_json(Layer4)
| extend DstPort = coalesce(tostring(Layer4Data.TCP.destination_port), tostring(Layer4Data.UDP.destination_port))
| project TimeGenerated, SrcIP, DstIP, DstPort, Verdict
| order by TimeGenerated desc
| take 20
```
**What you'll discover:**

> Exact source IPs of blocked external users (203.0.113.45 = your test machine)
> Destination port 80 confirms HTTP traffic is being blocked
> All INGRESS traffic to store-front is DROPPED - complete outage for external users

*Root cause for INGRESS:* The chaos policy has ingress: - fromEndpoints: [] (allow from nowhere = block all ingress)

**Query 3:** *Diagnose EGRESS - Separate DNS Failures from Connection Failures*

* Let's look at *DNS traffic (port 53)* to understand which domains are allowed vs blocked:

```
ContainerNetworkLogs
| where TimeGenerated > ago(30m)
| where SourceNamespace == "pets"
| where SourcePodName contains "store-front"
| where TrafficDirection == "EGRESS"
| extend Layer4Data = parse_json(Layer4)
| extend DstPort = coalesce(tostring(Layer4Data.TCP.destination_port), tostring(Layer4Data.UDP.destination_port))
| where DstPort == "53"  // DNS port
| summarize 
    Count = count()
    by Verdict
| order by Verdict asc
```

**What you'll discover:**

*Cumulative insights*

> DNS queries succeeded (microsoft.com, internal cluster DNS)
> DNS queries blocked (api.github.com, google.com, bing.com)
> Pattern identified: DNS is being selectively filtered

**Key insight:** Some domains fail at the DNS level - they never even get to attempt the HTTPS connection. This suggests the DNS `matchPattern` rules are too restrictive.

**Query 4:** *Complete Traffic Pattern - DNS + HTTPS Correlation*

* Now let's correlate DNS queries with HTTPS connection attempts to understand the full flow:

```
ContainerNetworkLogs
| where TimeGenerated > ago(30m)
| where SourceNamespace == "pets"
| where SourcePodName contains "store-front"
| where TrafficDirection == "EGRESS"
| extend SrcIP = tostring(IP.source), DstIP = tostring(IP.destination)
| extend Layer4Data = parse_json(Layer4)
| extend DstPort = coalesce(tostring(Layer4Data.TCP.destination_port), tostring(Layer4Data.UDP.destination_port))
| where DstPort in ("53", "443")  // DNS and HTTPS
| where isnotempty(DstIP)
| project TimeGenerated, DstIP, DstPort, Verdict
| order by TimeGenerated asc
| take 50
```

**What you'll discover:**


*Cumulative insights:*

**For microsoft.com (✅ Working):**

* Step 1: DNS query to 10.0.0.10 port 53 → FORWARDED
* Step 2: HTTPS connection to 23.192.18.101 port 443 → FORWARDED
* Result: Network allows traffic (HTTP 403 from server is expected)

**For api.github.com, google.com, bing.com (❌ Failing):**

* Step 1: DNS query to 10.0.0.10 port 53 → DROPPED
* Step 2: HTTPS connection → Never attempted (DNS failed first)
* Result: "bad address" error

*Root cause confirmed:* The FQDN policy is missing these domains from the DNS matchPattern rules. Even though api.github.com is in the toFQDNs list, the DNS query gets blocked first.

*Educational insight:* This demonstrates a common FQDN policy misconfiguration - you need BOTH DNS rules AND toFQDNs rules for external access to work.

**Query 6:** *Complete Diagnosis*

* Get a comprehensive view of all traffic patterns to confirm your diagnosis:

```
ContainerNetworkLogs
| where TimeGenerated > ago(30m)
| where SourceNamespace == "pets" or DestinationNamespace == "pets"
| summarize
    TotalFlows = count(),
    DroppedFlows = countif(Verdict == "DROPPED"),
    ForwardedFlows = countif(Verdict == "FORWARDED")
    by TrafficDirection, SourcePodName, DestinationPodName
| extend DropRate = round((DroppedFlows * 100.0) / TotalFlows, 2)
| where TotalFlows > 5  // Filter out noise
| order by DroppedFlows desc
| take 20
```

**What you'll discover:**

* *Evidence:* EGRESS to external has 25.8% drop rate, DNS queries have 26.7% drop rate
* *Root Cause:* FQDN policy missing domains (api.github.com, google.com, bing.com) from DNS matchPattern rules
* *Fix Needed:* Add missing domains to DNS patterns OR remove from toFQDNs list

**Problem 3:** Internal Communication Working Fine

* *Evidence:* EGRESS between pods in pets namespace has greater than 5% drop rate (filtered out)
* *Conclusion:* Application code and internal services are healthy

### Optimize Metrics Collection with Container Network Metrics Filtering

* Instead of collecting everything and filtering later (expensive).
* We define what matters at the source (cost-effective).


### How metrics filtering works

* Metrics filtering uses a Kubernetes **Custom Resource Definition (CRD)** called `ContainerNetworkMetric`.
* You define filtering rules that tell the Cilium agents which Hubble metrics to collect and which to skip.

**Key characteristics:**

* **One CRD per cluster** — only a single `ContainerNetworkMetric` resource can exist
* **Dynamic reconciliation** — changes take effect in approximately 30 seconds without restarting Cilium agents or Prometheus
* **No CRD = collect everything** — if you don't apply a CRD, all metrics are collected (the current default behavior)
* **Four metric types** — `dns`, `flow`, `tcp`, and `drop`
* **Include and exclude filters** — target by namespace/pod, label selector, IP/CIDR, protocol, or verdict

### Apply a targeted metrics filter for the pets namespace

* apply a filter that focuses metrics collection on the `pets` namespace (where your application workloads run) and excludes noisy system namespaces.

```yaml
kubectl apply -f - <<'EOF'
apiVersion: acn.azure.com/v1alpha1
kind: ContainerNetworkMetric
metadata:
  name: container-network-metric
spec:
  filters:
    # Collect flow metrics only for the pets namespace
    - metric: flow
      includeFilters:
        - name: pets-app-flows
          from:
            labelSelector:
              matchExpressions:
                - key: k8s.io/namespace
                  operator: In
                  values:
                    - pets
        - name: pets-app-flows-ingress
          to:
            labelSelector:
              matchExpressions:
                - key: k8s.io/namespace
                  operator: In
                  values:
                    - pets

    # Collect DNS metrics but exclude kube-system noise
    - metric: dns
      excludeFilters:
        - name: exclude-system-dns
          from:
            namespacedPod:
              - "kube-system/"

    # Collect TCP metrics only for the pets namespace
    - metric: tcp
      includeFilters:
        - name: pets-tcp-metrics
          from:
            labelSelector:
              matchExpressions:
                - key: k8s.io/namespace
                  operator: In
                  values:
                    - pets

    # Collect all drop metrics (always useful for troubleshooting)
    - metric: drop
      excludeFilters:
        - name: exclude-system-drops
          from:
            namespacedPod:
              - "kube-system/"
EOF
```

### Verify the CRD was applied:

```bash
kubectl get ContainerNetworkMetric
```
You should see your `container-network-metric` resource listed in 30 seconds.

**Understanding the filter**

| Metric type | Filter strategy | What it collects | What it skips |
|---|---|---|---|
| `flow` | Include only `pets` namespace | Pod-to-pod flows for store-front, order-service, product-service, etc. | All kube-system, monitoring, and other namespace flows |
| `dns` | Exclude `kube-system` | DNS queries from `pets` and other app namespaces | CoreDNS internal metrics, system DNS chatter |
| `tcp` | Include only `pets` namespace | TCP connection states for your application | System-level TCP connections |
| `drop` | Exclude `kube-system` | Dropped packets in application namespaces (useful for policy debugging) | Expected system-level drops |

| Part | Meaning |
| --- | --- |

| `--timeout=2` | Stop trying if no response is received within 2 seconds |
| `http://product-service:3002/` | The internal service URL/port to test |
| `2>/dev/null` | Redirect error messages (standard error) to nowhere (silence errors) |

### Command Breakdown



| Part | Meaning |
| --- | --- |
| `kubectl exec -n pets -it` | Run a command interactively in a pod within the `pets` namespace |
| `$(kubectl get po ...)` | Dynamically fetch the name of the `order-service` pod to use as the target |

### Experiment with different filter strategies

**Strategy A: Monitor only dropped packets (minimum cost)**

```yaml
kubectl apply -f - <<'EOF'
apiVersion: acn.azure.com/v1alpha1
kind: ContainerNetworkMetric
metadata:
  name: container-network-metric
spec:
  filters:
    - metric: drop
      includeFilters:
        - name: all-drops
          verdict:
            - dropped
EOF
```

* This is the most aggressive cost reduction — you only collect metrics for dropped packets, which are the most actionable for troubleshooting.

---

**Strategy B: Production vs. non-production filtering**

```YAML
kubectl apply -f - <<'EOF'
apiVersion: acn.azure.com/v1alpha1
kind: ContainerNetworkMetric
metadata:
  name: container-network-metric
spec:
  filters:
    - metric: flow
      includeFilters:
        - name: production-only
          from:
            labelSelector:
              matchLabels:
                environment: production
    - metric: dns
      includeFilters:
        - name: production-dns
          from:
            labelSelector:
              matchLabels:
                environment: production
    - metric: tcp
      includeFilters:
        - name: production-tcp
          from:
            labelSelector:
              matchLabels:
                environment: production
    - metric: drop
      includeFilters:
        - name: all-env-drops
          verdict:
            - dropped
EOF
```
